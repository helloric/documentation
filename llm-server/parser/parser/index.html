    Using the Parser

    Calling the Parser

    There are two ways to call the Parser. Either by giving it a Lexer or by giving it an input string.

    from parsing.lexer import Lexer
from parsing.parser import Parser

# ...

parser = Parser(Lexer('BEGIN <HAPPY> "Hi mom!" END'))
expressions = parser.parse()

    from parsing.parser import Parser

# ...

parser = Parser('BEGIN <HAPPY> "Hi mom!" END')
expressions = parser.parse()

    This is purely created for ease-of-use whilst keeping the option open to pass the Lexer as an argument. There are no differences between these two options.

    You can furthermore set parameters for the result analysis, if the default values don't suit you.

    from parsing.parser import Parser

# ...

parser = Parser('BEGIN <HAPPY> "Hi mom!" END', max_words_between_pause=5, max_emotions_between_pause=2)
parser.parse(result_analysis=True)

    With the function parse, you can begin the parsing process.

    If you'd like to perform a result analysis after parsing, you have to pass the argument result_analysis=True for that.

    Expanding the Parser

    Currently, the Parser is built such that every method is either a helper method for handling tokens or represents a Non-Terminal of our EBNF-Grammar.

    Info

    For more information about the EBNF (Extended Backus-Naur form), there is a wikipedia article that explains it.

    For instance, _emotion is the method that represents the production rule for the Emotion Non-Terminal. Every Non-Terminal has a production rule, if it is used.

    If we call _emotion, we expect to consume Tokens in the order that the production rule is displaying those Tokens in. We save the data we need from those Tokens.

    Changing the Grammar

    Before we add any new code to the Parser, let's first define what we wanna have.

    Suppose we want to expand on the square brackets idea from the Lexer tutorial. Those square brackets should now be used to create an emphasis on some text. Let's change the Grammar first.

    Start ::= 'BEGIN' Emotion (Highlight | Literal) { [';'] Emotion (Highlight | Literal) }
Highlight ::= '[' Literal ']'

    Updating the Parser

    Now that we have new rules, we need to update them in the Parser.

    def parse(self, result_analysis: bool = False) -> List[Expression]:
    self.lexer.next_token()
    self._expect(Id.BEGIN)
    emotion = self._emotion()

    # Lookahead if we expect a square-bracket first or if it's just a literal.
    expression : Expression
    if self._lookahead(Id.LSQBRACKET):
        expression = self._highlight(emotion)
    else:
        expression = SpeechExpression(emotion, self._literal())

    sequence : list[Expression] = [expression]

    while not self._lookahead(Id.END) and not self._lookahead(Id.EOF):    
        if self._lookahead(Id.SEMICOLON):
            self.lexer.next_token()
            sequence.append(PauseExpression())
        emotion = self._emotion()
        # Same thing here. Check if we expect a square-bracket first or if it's just a literal
        if self._lookahead(Id.LSQBRACKET):
            sequence.append(self._highlight(emotion))
        else:
            sequence.append(SpeechExpression(emotion, self._literal()))
    self._expect(Id.END)

    if result_analysis:
        self._result_analysis(sequence)

    return sequence

    Next, we need to actually create the rule we'd like to add So we add a new function in Parser.

    def _highlight(self, emotion: Emotion) -> HighlightExpression:
    self._expect(Id.LSQBRACKET)
    text = self._literal()
    self._expect(Id.RSQBRACKET)
    return HighlightExpression(emotion, text)

    Adding a new Expression

    Now we have our rules, but we also need to define a new data structure for it.

    Warning

    This is only done for demonstrative purposes. We could easily add a bool variable in SpeechExpression and it'd do the same trick. However, this showcases how to add new Expressions.

    Info

    For more info on what expressions are, check out the definition of already existing expressions.

    We go into the expression.py. Here, we can already see a few Expressions that have been in use before.

    We go below all of the expressions and add a new class.

    # ...
class HighlightExpression(Expression):
    def __init__(self, emotion: Emotion, text: str):
        self.emotion = emotion
        self.text = text

    def __eq__(self, value: object) -> bool:
        return super().__eq__(value) and cast(value, HighlightExpression).emotion == emotion and cast(value, HighlightExpression).text == text

    Note

    The __eq__ definition is important for Unit Testing later on, as a helper function there works by comparing two expressions with __eq__.

    Now, we have our new class and the data we want in that class. The Parser in our example should now work flawlessly again.

    Updating the Result Analysis

    The Result analysis is currently only designed to handle Pauses and SpeechExpressions. Our HighlightExpression is not yet handled.

    For instance, we could just add another case for HighlightExpressions:

    def _result_analysis(self, sequence: List[Expression]):
    emotion_stack = []
    word_count = 0
    for i in range(0, len(sequence)):
        expr = sequence[i]
        if isinstance(expr, PauseExpression):
            emotion_stack.clear()
            word_count = 0
            continue
        if len(emotion_stack) > self.max_emotions_between_pause:
            raise Exception('Du änderst deine Emotionen viel zu häufig zwischen Pausen.')
        if isinstance(expr, SpeechExpression):
            expr = cast(SpeechExpression, expr)
            word_count += len(expr.text.split(' '))
            if word_count > self.max_words_between_pause:
                raise Exception('Du benutzt zu viele Wörter zwischen Pausen. Pausiere ab und zu deine Sprache.')
            emotion_stack.append(expr.emotion)
        elif isinstance(expr, HighlightExpression): # we start appending here
            expr = cast(HighlightExpression, expr)
            word_count += len(expr.text.split(' '))
            if word_count > self.max_words_between_pause:
                raise Exception('Du benutzt zu viele Wörter zwischen Pausen. Pausiere ab und zu deine Sprache.')

    Warning

    Again, for our case, it probably would've been better to just inherit from SpeechExpression itself or add another attribute to it. But this tutorial was to show how to add new rules and new Expressions.

    _result_analysis(sequence)

    Performs a result analysis on the speech sequence.

    A result analysis consists of checking certain data between PauseExpressions.

    We currently track two things:

    • Count of emotions between pauses

    • Count of words between pauses.

    If a certain threshold, that is set in the constructor, gets overstepped, this method raises an Exception, telling the LLM how to recalculate the answer.

    Parameters:

    Name Type Description Default
    sequence List[Expression]

    The sequence to perform a result analysis on.

    		 required

    Example

    >>> parser = Parser('BEGIN <HAPPY> "I love my life so much and I can\'t stop yapping about how much I love my life." <FRUSTRATED> "Woah!" END')
>>> result = parser.parse(result_analysis=True)
Error: Too many words between pauses.

    >>> parser = Parser('BEGIN <HAPPY> "Can\'t" <FRUSTRATED> "decide" <BORED> "how" <CALM> "I" <SAD> "feel!" END')
>>> result = parser.parse(result_analysis=True)
Error: Too many emotions between pauses.
